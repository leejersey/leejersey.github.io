# 第六章 消息队列与发布订阅

> 本章目标：掌握 Redis 三种消息机制——List 简单队列、Pub/Sub 实时广播、Stream 可靠消费者组，理解各自的适用边界。

---

## 6.1 List 实现简单任务队列

### 生产者-消费者模型

用 List 的 LPUSH + BRPOP 就能实现最基本的消息队列：

```
生产者 ──LPUSH──→ [ task3, task2, task1 ] ──BRPOP──→ 消费者
                       Redis List
                     （先进先出）
```

### 生产者

```python
import redis
import json
import time

r = redis.Redis(host="127.0.0.1", port=6379, db=0, decode_responses=True)

QUEUE_NAME = "queue:email"


def enqueue(queue: str, task: dict):
    """发送任务到队列"""
    message = json.dumps(task, ensure_ascii=False)
    r.lpush(queue, message)
    print(f"[Producer] 发送: {task}")


# 模拟发送邮件任务
enqueue(QUEUE_NAME, {"to": "alice@test.com", "subject": "欢迎注册", "type": "welcome"})
enqueue(QUEUE_NAME, {"to": "bob@test.com", "subject": "密码重置", "type": "reset"})
enqueue(QUEUE_NAME, {"to": "carol@test.com", "subject": "订单确认", "type": "order"})
```

### 消费者

```python
def consume(queue: str, handler, timeout: int = 30):
    """消费队列（阻塞模式）"""
    print(f"[Consumer] 开始监听 {queue}...")
    while True:
        result = r.brpop(queue, timeout=timeout)
        if result is None:
            print("[Consumer] 等待超时，继续监听...")
            continue

        _, message = result
        task = json.loads(message)
        print(f"[Consumer] 收到: {task}")

        try:
            handler(task)
        except Exception as e:
            print(f"[Consumer] 处理失败: {e}")
            # 失败任务放入死信队列
            r.lpush(f"{queue}:dead", message)


def send_email(task: dict):
    """模拟发送邮件"""
    print(f"  → 发送邮件到 {task['to']}: {task['subject']}")
    time.sleep(0.5)  # 模拟耗时


# 启动消费者（会阻塞）
# consume(QUEUE_NAME, send_email)
```

### 可靠队列：RPOPLPUSH 模式

普通的 BRPOP 有个问题：消费者拿到消息后如果崩溃了，**消息就丢了**。

RPOPLPUSH 把消息从工作队列**原子地转移到处理中队列**，处理完再删除：

```python
PROCESSING_QUEUE = f"{QUEUE_NAME}:processing"


def consume_reliable(queue: str, handler, timeout: int = 30):
    """可靠消费：处理失败可恢复"""
    while True:
        # 原子地从 queue 弹出并放入 processing 队列
        message = r.brpoplpush(queue, PROCESSING_QUEUE, timeout=timeout)
        if message is None:
            continue

        task = json.loads(message)
        try:
            handler(task)
            # 处理成功，从 processing 队列删除
            r.lrem(PROCESSING_QUEUE, 1, message)
        except Exception as e:
            print(f"处理失败: {e}")
            # 消息留在 processing 队列，可以后续重试或人工处理


def recover_stuck_messages(max_age: int = 60):
    """恢复卡在 processing 队列中的消息（消费者崩溃恢复）"""
    messages = r.lrange(PROCESSING_QUEUE, 0, -1)
    for msg in messages:
        # 放回主队列重新消费
        r.lrem(PROCESSING_QUEUE, 1, msg)
        r.lpush(QUEUE_NAME, msg)
        print(f"[Recovery] 已恢复: {msg}")
```

### List 队列的局限

| 能力 | List 队列 |
|------|-----------|
| 基本生产消费 | ✅ |
| 阻塞消费 | ✅ BRPOP |
| 消息确认（ACK） | ❌ 需自己实现 |
| 消费者组 | ❌ 不支持 |
| 消息回溯 | ❌ 消费即删除 |
| 广播 | ❌ 只有一个消费者能拿到 |

> 需要广播 → 用 Pub/Sub；需要消费者组和 ACK → 用 Stream。

## 6.2 Pub/Sub：实时通知

### 发布/订阅模型

Pub/Sub 是广播机制——**一条消息可以被多个订阅者同时收到**：

```
Publisher ──PUBLISH──→ Channel "news" ──→ Subscriber A
                                     ──→ Subscriber B
                                     ──→ Subscriber C
```

### 发布者

```python
def publish_notification(channel: str, message: dict):
    """发布消息"""
    data = json.dumps(message, ensure_ascii=False)
    count = r.publish(channel, data)
    print(f"[Pub] 发送到 {channel}，{count} 个订阅者收到")

# 发布
publish_notification("notifications:order", {
    "event": "order_paid",
    "order_id": 1001,
    "amount": 99.9,
})
```

### 订阅者

```python
def subscribe_channel(channels: list[str], handler):
    """订阅频道"""
    pubsub = r.pubsub()
    pubsub.subscribe(*channels)

    print(f"[Sub] 订阅频道: {channels}")
    for message in pubsub.listen():
        if message["type"] != "message":
            continue  # 跳过订阅确认消息

        channel = message["channel"]
        data = json.loads(message["data"])
        print(f"[Sub] 收到 [{channel}]: {data}")
        handler(channel, data)


def handle_order_event(channel: str, data: dict):
    """处理订单事件"""
    if data["event"] == "order_paid":
        print(f"  → 发送支付成功通知，订单: {data['order_id']}")

# 启动订阅（会阻塞）
# subscribe_channel(["notifications:order"], handle_order_event)
```

### 模式订阅

用通配符订阅多个频道：

```python
pubsub = r.pubsub()

# 订阅所有 notifications: 开头的频道
pubsub.psubscribe("notifications:*")

for message in pubsub.listen():
    if message["type"] != "pmessage":
        continue
    pattern = message["pattern"]    # 'notifications:*'
    channel = message["channel"]    # 'notifications:order'
    data = message["data"]
    print(f"匹配 {pattern}，频道 {channel}: {data}")
```

### 异步订阅

```python
import redis.asyncio as aioredis
import asyncio


async def async_subscriber():
    """异步订阅"""
    client = aioredis.Redis(host="127.0.0.1", port=6379, decode_responses=True)
    pubsub = client.pubsub()
    await pubsub.subscribe("notifications:chat")

    print("[AsyncSub] 开始监听...")
    async for message in pubsub.listen():
        if message["type"] != "message":
            continue
        data = json.loads(message["data"])
        print(f"[AsyncSub] 收到: {data}")

# asyncio.run(async_subscriber())
```

### Pub/Sub 的局限

| 特性 | 说明 |
|------|------|
| ❌ 无持久化 | 消息发出后不存储，离线的订阅者收不到 |
| ❌ 无 ACK | 不知道消息是否被成功处理 |
| ❌ 无回溯 | 不能查看历史消息 |
| ✅ 适合场景 | 实时通知、聊天室、配置变更广播 |

> Pub/Sub 的核心定位：**实时广播，不保证可靠交付**。需要可靠性请用 Stream。

## 6.3 Redis Stream：带消费者组的可靠队列

### Stream 是什么？

Redis 5.0 引入的 Stream 是一个**持久化的、支持消费者组的消息队列**——本质上是 Redis 版的 Kafka（轻量版）。

```
┌─────────────────────────────────────────────┐
│  Stream: "stream:orders"                     │
│                                              │
│  ID              │ Data                      │
│  1709000001-0    │ {order_id:1, amount:99}   │
│  1709000002-0    │ {order_id:2, amount:50}   │
│  1709000003-0    │ {order_id:3, amount:200}  │
│                                              │
│  ┌─── Consumer Group: "workers" ──────────┐ │
│  │  Consumer A: 已处理到 1709000002-0      │ │
│  │  Consumer B: 已处理到 1709000001-0      │ │
│  │  Pending: [1709000003-0 → Consumer A]   │ │
│  └─────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

### 写入消息

```python
# XADD：写入消息，* 表示自动生成 ID
msg_id = r.xadd("stream:orders", {
    "order_id": "1001",
    "amount": "99.9",
    "user_id": "2001",
})
print(f"消息 ID: {msg_id}")  # '1709000001234-0'

# 批量写入
for i in range(1, 6):
    r.xadd("stream:orders", {
        "order_id": str(1000 + i),
        "amount": str(i * 10),
    })
```

### 读取消息（不用消费者组）

```python
# 读取最新 5 条
messages = r.xrevrange("stream:orders", count=5)
for msg_id, data in messages:
    print(f"{msg_id}: {data}")

# 阻塞等待新消息（类似 BRPOP）
# '$' 表示只接收新消息
messages = r.xread({"stream:orders": "$"}, block=5000, count=1)
# block=5000 → 最多等 5 秒
```

### 消费者组：核心功能

消费者组让多个消费者**协同消费**同一个 Stream，每条消息只会被组内一个消费者处理：

```python
STREAM = "stream:orders"
GROUP = "order_workers"

# 创建消费者组（0 表示从头开始消费，$ 表示只消费新消息）
try:
    r.xgroup_create(STREAM, GROUP, id="0", mkstream=True)
    print(f"创建消费者组: {GROUP}")
except redis.ResponseError as e:
    if "BUSYGROUP" in str(e):
        print(f"消费者组 {GROUP} 已存在")
    else:
        raise
```

### 消费者

```python
def stream_consumer(stream: str, group: str, consumer_name: str):
    """Stream 消费者"""
    print(f"[{consumer_name}] 开始消费 {stream} (组: {group})")

    while True:
        # 读取分配给自己的消息
        # '>' 表示读取未消费的新消息
        messages = r.xreadgroup(
            groupname=group,
            consumername=consumer_name,
            streams={stream: ">"},
            count=5,       # 每次最多读 5 条
            block=5000,    # 最多等 5 秒
        )

        if not messages:
            continue

        for stream_name, msg_list in messages:
            for msg_id, data in msg_list:
                print(f"[{consumer_name}] 处理: {msg_id} → {data}")

                try:
                    process_order(data)
                    # 处理成功，确认消息（ACK）
                    r.xack(stream, group, msg_id)
                    print(f"[{consumer_name}] ACK: {msg_id}")
                except Exception as e:
                    print(f"[{consumer_name}] 处理失败: {e}")
                    # 不 ACK → 消息留在 Pending 列表，可以重新消费


def process_order(data: dict):
    """处理订单"""
    print(f"  → 处理订单 {data.get('order_id')}，金额 {data.get('amount')}")
    time.sleep(0.3)


# 启动消费者（通常在不同的进程/线程中）
# stream_consumer(STREAM, GROUP, "worker-1")
# stream_consumer(STREAM, GROUP, "worker-2")
```

### 处理 Pending 消息（崩溃恢复）

消费者崩溃后，它未 ACK 的消息会留在 Pending 列表。可以由其他消费者接管：

```python
def recover_pending(stream: str, group: str, consumer_name: str, min_idle_ms: int = 60000):
    """恢复超时未 ACK 的 Pending 消息"""
    # 查看 Pending 消息
    pending = r.xpending_range(stream, group, "-", "+", count=10)

    for item in pending:
        msg_id = item["message_id"]
        idle_time = item["time_since_delivered"]
        original_consumer = item["consumer"]

        if idle_time > min_idle_ms:
            # 超过 1 分钟未 ACK，转移给当前消费者
            claimed = r.xclaim(
                stream, group, consumer_name,
                min_idle_time=min_idle_ms,
                message_ids=[msg_id],
            )
            for claimed_id, data in claimed:
                print(f"[Recovery] 接管消息 {claimed_id} (原: {original_consumer})")
                process_order(data)
                r.xack(stream, group, claimed_id)
```

```python
info = r.xinfo_stream(STREAM)
print(f"消息总数: {info['length']}")
print(f"第一条: {info['first-entry']}")

```

### 三种方案对比

| | List 队列 | Pub/Sub | Stream |
|---|---|---|---|
| **持久化** | ✅ | ❌ | ✅ |
| **消费确认 (ACK)** | ❌ 需自实现 | ❌ | ✅ 内置 |
| **消费者组** | ❌ | ❌ | ✅ 内置 |
| **消息回溯** | ❌ 消费即删 | ❌ | ✅ 可回看 |
| **广播** | ❌ 单消费者 | ✅ | ✅ 多消费者组 |
| **复杂度** | 低 | 低 | 中 |
| **适合** | 简单异步任务 | 实时通知 | 可靠消息队列 |

> 📌 **选择建议**：
> - 简单异步任务（如发邮件）→ **List**
> - 实时广播（如聊天、通知）→ **Pub/Sub**
> - 需要可靠消费、多消费者→ **Stream**
> - 对可靠性要求极高 → 用专业 MQ（RabbitMQ / Kafka）

## 本章小结

| 知识点 | 要点 |
|--------|------|
| List 队列 | LPUSH + BRPOP，简单但无 ACK |
| 可靠队列 | BRPOPLPUSH + processing 队列 |
| Pub/Sub | 实时广播，无持久化，离线丢消息 |
| Stream | XADD/XREADGROUP/XACK，Redis 版 Kafka |
| 消费者组 | 多消费者协同，每条消息只处理一次 |
| Pending 恢复 | XCLAIM 接管超时消息 |
| XTRIM | 限制 Stream 长度，防无限增长 |
