## 9.1 Key 命名规范与 TTL 策略

### 好的 Key 命名 = 好的运维体验

Redis 没有表、没有 Schema，**Key 就是你唯一的组织手段**。命名混乱的 Redis 就像没有文件夹的电脑桌面——数据越多越崩溃。

### 命名规范

```
格式：{业务}:{对象}:{标识}[:子字段]

✅ 好的命名：
  cache:user:1001              → 用户 1001 的缓存
  cache:article:2001           → 文章 2001 的缓存
  session:a1b2c3d4             → 会话数据
  lock:order:1001              → 订单 1001 的分布式锁
  ratelimit:api:login:10.0.1.5 → 登录接口对 IP 的限流
  queue:email                  → 邮件发送队列
  rank:game:daily              → 每日游戏排行榜

❌ 坏的命名：
  user1001        → 没有分隔，无法按前缀扫描
  u:1001          → 缩写太短，含义不明
  UserCache_1001  → 大小写混乱，下划线和冒号混用
  test123         → 没有业务含义
```

### 用常量管理 Key

```python
class RedisKeys:
    """Redis Key 常量（避免魔法字符串散落各处）"""

    # 缓存
    CACHE_USER = "cache:user:{user_id}"
    CACHE_ARTICLE = "cache:article:{article_id}"

    # 会话
    SESSION = "session:{session_id}"

    # 锁
    LOCK_ORDER = "lock:order:{order_id}"

    # 限流
    RATE_LIMIT = "ratelimit:{endpoint}:{identifier}:{window}"

    # 队列
    QUEUE_EMAIL = "queue:email"
    QUEUE_NOTIFICATION = "queue:notification"

    # 排行榜
    RANK_DAILY = "rank:{category}:daily:{date}"

    @staticmethod
    def cache_user(user_id: int) -> str:
        return RedisKeys.CACHE_USER.format(user_id=user_id)

    @staticmethod
    def lock_order(order_id: int) -> str:
        return RedisKeys.LOCK_ORDER.format(order_id=order_id)


# 使用
key = RedisKeys.cache_user(1001)  # "cache:user:1001"
```

### TTL 策略

| 数据类型 | 建议 TTL | 理由 |
|----------|----------|------|
| 接口响应缓存 | 5-60 分钟 | 数据变化快，TTL 不宜太长 |
| 用户信息缓存 | 1-24 小时 | 变化较少 |
| 配置缓存 | 1-7 天 | 几乎不变 |
| Session | 7-30 天 | 用户体验 |
| 验证码 | 5-10 分钟 | 安全性 |
| 分布式锁 | 10-60 秒 | 防死锁 |
| 限流计数器 | 窗口大小 +1s | 自动清理 |

```python
class TTL:
    """TTL 常量（秒）"""
    CACHE_SHORT = 5 * 60         # 5 分钟
    CACHE_MEDIUM = 60 * 60       # 1 小时
    CACHE_LONG = 24 * 60 * 60    # 1 天
    SESSION = 7 * 24 * 60 * 60   # 7 天
    VERIFICATION = 5 * 60        # 5 分钟
    LOCK = 10                    # 10 秒
```

> 📌 **铁律：每个 Key 都必须有 TTL**。没有 TTL 的 Key 会永远留在内存中，最终导致 OOM。

## 9.2 序列化：JSON vs MessagePack vs Pickle

### 为什么要关心序列化？

Redis 的 Value 只能存字符串或字节。Python 对象（dict、list、dataclass）需要序列化后才能存入，读取时再反序列化：

```
Python dict → 序列化 → Redis String → 反序列化 → Python dict
```

### 三种方案对比

```python
import json
import pickle
import msgpack  # pip install msgpack

data = {
    "id": 1001,
    "name": "张三",
    "tags": ["python", "redis", "fastapi"],
    "score": 95.5,
    "active": True,
}

# JSON
json_bytes = json.dumps(data, ensure_ascii=False).encode()
print(f"JSON:     {len(json_bytes)} bytes")  # ~110 bytes

# MessagePack
msgpack_bytes = msgpack.packb(data, use_bin_type=True)
print(f"MsgPack:  {len(msgpack_bytes)} bytes")  # ~75 bytes

# Pickle
pickle_bytes = pickle.dumps(data)
print(f"Pickle:   {len(pickle_bytes)} bytes")  # ~140 bytes
```

| | JSON | MessagePack | Pickle |
|---|---|---|---|
| **体积** | 中 | 小（省 30-40%） | 大 |
| **速度** | 快 | 更快 | 中 |
| **可读性** | ✅ 人类可读 | ❌ 二进制 | ❌ 二进制 |
| **跨语言** | ✅ 通用 | ✅ 多语言支持 | ❌ Python 专属 |
| **安全性** | ✅ 安全 | ✅ 安全 | ⚠️ 反序列化可执行代码 |
| **类型支持** | 基础类型 | 基础类型 | 任意 Python 对象 |

### 封装统一序列化器

```python
from abc import ABC, abstractmethod
from enum import Enum


class Serializer(ABC):
    @abstractmethod
    def dumps(self, obj) -> bytes: ...
    @abstractmethod
    def loads(self, data: bytes): ...


class JsonSerializer(Serializer):
    def dumps(self, obj) -> bytes:
        return json.dumps(obj, ensure_ascii=False, default=str).encode()

    def loads(self, data: bytes):
        return json.loads(data)


class MsgpackSerializer(Serializer):
    def dumps(self, obj) -> bytes:
        return msgpack.packb(obj, use_bin_type=True)

    def loads(self, data: bytes):
        return msgpack.unpackb(data, raw=False)


# 工厂
def get_serializer(name: str = "json") -> Serializer:
    serializers = {
        "json": JsonSerializer,
        "msgpack": MsgpackSerializer,
    }
    return serializers[name]()


# 在缓存服务中使用
class CacheService:
    def __init__(self, redis, serializer: Serializer = None):
        self.redis = redis
        self.serializer = serializer or JsonSerializer()

    async def get(self, key: str):
        data = await self.redis.get(key)
        if data is None:
            return None
        return self.serializer.loads(data if isinstance(data, bytes) else data.encode())

    async def set(self, key: str, value, ttl: int = 3600):
        data = self.serializer.dumps(value)
        await self.redis.set(key, data, ex=ttl)
```

> 📌 **选择建议**：
> - 默认用 **JSON**（可读、安全、通用）
> - 数据量大且对体积敏感 → **MessagePack**
> - **永远不用 Pickle**（安全风险：反序列化时可执行任意代码）

## 9.3 异常处理、超时与降级策略

### 核心原则：缓存允许失败

Redis 是缓存层，不是核心数据源。**Redis 挂了不应该导致整个服务不可用**。

```
✅ 正确心态：Redis 只是加速器
  Redis 可用 → 走缓存，快
  Redis 不可用 → 跳过缓存，直接查 DB，慢但能用

❌ 错误心态：Redis 是必需品
  Redis 不可用 → 抛异常 → 500 错误 → 用户看到错误页面
```

### 超时配置

```python
import redis.asyncio as aioredis

# 为连接和读写分别设置超时
pool = aioredis.ConnectionPool(
    host="127.0.0.1",
```

### 优雅降级装饰器

```python
import functools
import logging

logger = logging.getLogger(__name__)


def cache_graceful(fallback=None):
    """
    缓存优雅降级装饰器：Redis 出错时不影响业务

    Args:
        fallback: 降级时的返回值（默认 None）
    """
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            try:
                return await func(*args, **kwargs)
            except (
                redis.ConnectionError,
                redis.TimeoutError,
                redis.RedisError,
            ) as e:
                logger.warning(f"Redis 操作失败，已降级: {func.__name__} - {e}")
                return fallback
        return wrapper
    return decorator


# 使用
@cache_graceful(fallback=None)
async def get_cached_user(redis_client, user_id: int):
    data = await redis_client.get(f"cache:user:{user_id}")
    return json.loads(data) if data else None

@cache_graceful(fallback=False)
async def set_cached_user(redis_client, user_id: int, user_data: dict):
    await redis_client.set(
        f"cache:user:{user_id}",
        json.dumps(user_data),
        ex=3600,
    )
    return True
```

### 降级后的完整读取流程

```python
async def get_user(user_id: int, redis_client, db_session) -> dict | None:
    """完整的读取流程：缓存 → DB，缓存失败不影响业务"""

    # 尝试从缓存读取
    cached = await get_cached_user(redis_client, user_id)
    if cached is not None:
        return cached

    # 缓存未命中（或 Redis 不可用），查数据库
    user = await query_user_from_db(db_session, user_id)

    if user:
        # 尝试回写缓存（失败也没关系）
        await set_cached_user(redis_client, user_id, user)

    return user
```

### 断路器模式（进阶）

当 Redis 连续失败多次时，暂时停止尝试（避免浪费时间等超时）：

```python
import time


class CircuitBreaker:
    """简易断路器"""

    def __init__(self, failure_threshold: int = 5, recovery_timeout: int = 30):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.failures = 0
        self.last_failure_time = 0
        self.state = "closed"  # closed(正常) / open(熔断) / half_open(试探)

    def can_execute(self) -> bool:
        if self.state == "closed":
            return True
        if self.state == "open":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "half_open"
                return True  # 允许一次试探
            return False
        return True  # half_open

    def record_success(self):
        self.failures = 0
        self.state = "closed"

    def record_failure(self):
        self.failures += 1
        self.last_failure_time = time.time()
        if self.failures >= self.failure_threshold:
            self.state = "open"
            logger.warning(f"断路器打开：连续失败 {self.failures} 次，暂停 {self.recovery_timeout}s")


redis_breaker = CircuitBreaker(failure_threshold=5, recovery_timeout=30)


async def safe_redis_get(redis_client, key: str):
    """带断路器的 Redis 读取"""
    if not redis_breaker.can_execute():
        return None  # 熔断中，直接跳过

    try:
        result = await redis_client.get(key)
        redis_breaker.record_success()
        return result
    except (redis.ConnectionError, redis.TimeoutError):
        redis_breaker.record_failure()
        return None
```

## 9.4 监控：内存、命中率、慢查询

### 基本健康检查

```python
@app.get("/health/redis")
async def redis_health(redis: aioredis.Redis = Depends(get_redis)):
    """Redis 健康检查端点"""
    try:
        info = await redis.info(section="server")
        memory = await redis.info(section="memory")
        stats = await redis.info(section="stats")
        keyspace = await redis.info(section="keyspace")

        return {
            "status": "healthy",
            "version": info.get("redis_version"),
            "uptime_days": info.get("uptime_in_days"),
            "memory": {
                "used_mb": round(memory.get("used_memory", 0) / 1024 / 1024, 2),
                "peak_mb": round(memory.get("used_memory_peak", 0) / 1024 / 1024, 2),
                "fragmentation_ratio": memory.get("mem_fragmentation_ratio"),
            },
            "stats": {
                "connected_clients": info.get("connected_clients"),
                "total_commands": stats.get("total_commands_processed"),
                "hit_rate": _calc_hit_rate(stats),
            },
            "keys": keyspace,
        }
    except Exception as e:
        return {"status": "unhealthy", "error": str(e)}


def _calc_hit_rate(stats: dict) -> str:
    """计算缓存命中率"""
    hits = stats.get("keyspace_hits", 0)
    misses = stats.get("keyspace_misses", 0)
    total = hits + misses
    if total == 0:
        return "N/A"
    return f"{hits / total * 100:.1f}%"
```

### 关键指标

| 指标 | 命令 | 健康阈值 | 告警 |
|------|------|----------|------|
| **内存使用** | `INFO memory` | < maxmemory 的 80% | > 80% 告警 |
| **命中率** | `INFO stats` | > 90% | < 80% 需优化 |
| **连接数** | `INFO clients` | < maxclients 的 50% | > 80% 告警 |
| **慢查询** | `SLOWLOG GET` | 接近 0 | 持续增长需排查 |
| **内存碎片率** | `mem_fragmentation_ratio` | 1.0 ~ 1.5 | > 2.0 需重启 |

```python
async def get_slow_queries(redis: aioredis.Redis, count: int = 10) -> list:
    """获取最近的慢查询"""
    # 先设置慢查询阈值（微秒，10000 = 10ms）
    await redis.config_set("slowlog-log-slower-than", 10000)

    logs = await redis.slowlog_get(count)
    result = []
    for entry in logs:
        result.append({
            "id": entry.get("id"),
            "timestamp": entry.get("start_time"),
            "duration_ms": entry.get("duration", 0) / 1000,
            "command": " ".join(str(c) for c in entry.get("command", [])),
        })
    return result


@app.get("/debug/redis/slowlog")
async def slow_log(redis: aioredis.Redis = Depends(get_redis)):
    return {"slow_queries": await get_slow_queries(redis)}
```

### 定期健康报告

```python
async def redis_health_report(redis: aioredis.Redis):
    """定期输出 Redis 健康报告（可用 APScheduler 定时执行）"""
    memory = await redis.info(section="memory")
    stats = await redis.info(section="stats")
    clients = await redis.info(section="clients")

    used_mb = memory.get("used_memory", 0) / 1024 / 1024
    hit_rate = _calc_hit_rate(stats)
    connected = clients.get("connected_clients", 0)

    report = f"""
╔══════════════ Redis 健康报告 ══════════════╗
║ 内存使用:     {used_mb:.1f} MB
║ 命中率:       {hit_rate}
║ 连接数:       {connected}
║ 累计命令:     {stats.get('total_commands_processed', 0):,}
║ 过期淘汰:     {stats.get('evicted_keys', 0)}
╚════════════════════════════════════════════╝
"""
    logger.info(report)

    # 告警检查
    if used_mb > 500:
        logger.warning(f"⚠️ Redis 内存超过 500MB: {used_mb:.1f}MB")
    if hit_rate != "N/A" and float(hit_rate.rstrip('%')) < 80:
        logger.warning(f"⚠️ 缓存命中率过低: {hit_rate}")
```

## 本章小结

| 知识点 | 要点 |
|--------|------|
| Key 命名 | `{业务}:{对象}:{标识}` 格式，用常量类管理 |
| TTL | 每个 Key 都必须有 TTL，按数据类型选择合适时长 |
